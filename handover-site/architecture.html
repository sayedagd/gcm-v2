Architecture

Split system architecture

GCM v2 runs as two independently deployed applications: a static frontend and an API-only backend. They operate as one product through a clean HTTP API contract and explicit environment-based integration.

Top Level

Runtime boundaries

  • Frontend project: frontend/ on Vercel, static SPA build, browser-side routing.
  • Backend project: backend/ on Vercel, Express API routed through api/[...path].js.
  • Handover site: handover-site/ for local developer docs only.
  • Database: remote PostgreSQL accessed by backend via DATABASE_URL.

Request Flow

How frontend and backend act as one app

  1. Browser loads frontend static assets from the frontend Vercel project.
  2. Frontend API client reads VITE_API_BASE_URL and sends requests to backend Vercel URLs.
  3. Backend handles all business modules via Express routes (/api/*, /auth/* alias support).
  4. Backend validates JWT and role access, then performs PostgreSQL queries.
  5. Responses return to frontend for rendering and state updates.

Current Limitation

Storage note

Uploads/backups currently rely on filesystem behavior in backend services. For fully durable cloud behavior, move these paths to external object storage in a later hardening phase.
