Deployment

Build and release workflow

The deployment model is split: build the frontend as a static app, host the backend as an API service, and keep environment configuration outside source control.

Home Architecture Docs archive System knowledge Trips Reports Environment
  • Create two Vercel projects only: one rooted at frontend/ and one rooted at backend/.
  • Use the architecture reference page to verify service boundaries before first deploy.
  • Set VITE_API_BASE_URL in the frontend project to the backend Vercel URL.
  • Set backend env vars in Vercel: DATABASE_URL, JWT_SECRET, CORS_ORIGIN, plus required API keys.
  • Keep handover-site/ local-only for developer reference; do not deploy it to Vercel.
The old monolithic backend/frontend deployment model has been replaced by separate app targets with explicit CORS + API-base integration.
