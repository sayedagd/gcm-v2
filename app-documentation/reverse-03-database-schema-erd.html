## 2. Table Inventory

### Identity, Access, and Tenant Context

- `users`
  - PK: `id VARCHAR(100)`
  - Key fields: `email UNIQUE`, `password`, `role`, `company_id`, `project_id`, `supplier_id`, `status`

- `companies`
  - PK: `company_id VARCHAR(100)`
  - FK: `user_id -> users.id`
  - Key fields: registration, VAT, contact, branding/support files

- `saas_config`
  - PK: `id VARCHAR(50)` (singleton)
  - Key fields: multilingual app identity, landing/store/template/ai config blobs

### Core Planning and Service Model

- `projects`
  - PK: `project_id VARCHAR(100)`
  - FK: `company_id -> companies.company_id`, `user_id -> users.id`
  - Key fields: dates, budget, quantities, asset counts, status

- `services`
  - PK: `service_id VARCHAR(100)`
  - FK: `parent_id -> services.service_id`
  - Key fields: `major_category`, `requires_recycle_receipt`

- `project_services`
  - PK: `id VARCHAR(100)`
  - FK: `project_id -> projects.project_id`, `service_id -> services.service_id`
  - Key fields: quantity/price/cost/progress, supplier and warning thresholds

### Supplier, Facility, Fleet, and Workforce

- `suppliers`
  - PK: `supplier_id VARCHAR(100)`
  - FK: `user_id -> users.id`
  - Key fields: license/vat/bank/contract/status/project-service assignments

- `facilities`
  - PK: `facility_id VARCHAR(100)`
  - Key fields: type, contract, accepted services, location, status

- `drivers`
  - PK: `driver_id VARCHAR(100)`
  - FK: `user_id -> users.id`
  - Key fields: license/iqama/insurance card data, ownership, supplier/vehicle refs

- `vehicles`
  - PK: `vehicle_id VARCHAR(100)`
  - Unique: `plate_no`
  - Key fields: type, ownership, supplier, permit metadata

### Operations and Governance

- `trips`
  - PK: `trip_id VARCHAR(100)`
  - FK: `project_id`, `service_id`, `vehicle_id`, `driver_id`, `facility_id`
  - Key fields: quantity, status, lifecycle timestamps, compliance numbers/files, approvals

- `notifications`
  - PK: `id VARCHAR(100)`
  - FK: `user_id -> users.id`

- `activity_logs`
  - PK: `id VARCHAR(100)`
  - FK: `user_id -> users.id`

- `permission_requests`
  - PK: `id VARCHAR(100)`

- `contact_submissions`
  - PK: `id VARCHAR(100)`

### Inventory and Assets

- `inventory_sizes`
  - PK: `size_id VARCHAR(100)`

- `containers`
  - PK: `container_id VARCHAR(100)`
  - FK: `size_id`, `project_id`, `supplier_id`

- `tanks`
  - PK: `tank_id VARCHAR(100)`
  - FK: `size_id`, `project_id`, `supplier_id`

- `scales`
  - PK: `scale_id VARCHAR(100)`
  - FK: `size_id`, `project_id`, `supplier_id`

### Store and Inquiry Domain

- `environmental_equipments`
  - PK: `equipment_id VARCHAR(100)`
  - Key fields: multilingual descriptions, media links, specs, share count, availability

- `equipment_inquiries`
  - PK: `id VARCHAR(100)`
  - FK: `equipment_id -> environmental_equipments.equipment_id`
  - Key fields: customer contact, admin reply, status, timestamps

### Financial and AI Domain

- `project_supplier_rates`
  - PK: `id SERIAL`
  - FK: `project_id`, `supplier_id`, `service_id`
  - Key fields: `cost_price`, `currency`

- `ai_sessions`
  - PK: `id VARCHAR(100)`
  - FK: `user_id -> users.id`
  - Key fields: session lifecycle, role/language/action, confidence/rating/flags

- `ai_messages`
  - PK: `id VARCHAR(100)`
  - FK: `session_id -> ai_sessions.id`

## 3. Relationship and Integrity Notes

- Cascading deletes are used in planning/rate junction tables.
- Some foreign keys are `ON DELETE SET NULL` to preserve historical rows.
- Domain JSON payloads are stored as mixed `TEXT`/`JSONB` by table.
- Additional runtime tables are referenced by services (`backup_artifacts`, `backup_job_runs`) and should be treated as infrastructure metadata.
