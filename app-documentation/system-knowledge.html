Architecture and business rules

This is the operational source of truth for the platform: stack, monorepo structure, data flow, trip lifecycle, file storage, and the rules that must remain stable.

Home Docs archive Deployment Trips Driver lifecycle Reports
  • Frontend runtime: Next.js 16 App Router in frontend-next/; legacy Vite frontend is decommissioned.
  • Backend: Node.js + Express + PostgreSQL with modular domain routes and API v1 governance.
  • Contract governance: OpenAPI spec + generated frontend schema types checked during type-check workflows.
  • Frontend code organization: feature-bounded modules with enforced folder contract (ui/hooks/api/model/tests).
  • Endpoint normalization: typed operation mapping and endpoint constants centralized in core frontend feature modules.
  • Storage model: uploads and backups are object-storage-backed in production with signed read URLs.
  • Scheduler model: managed scheduler + idempotent/lock-protected system trigger endpoints.
  • Security posture: global write sanitization/body-shape guard, token-expiry re-auth contract, and endpoint-tier rate limits.
  • Performance posture: explicit route render/cache matrix, bundle budget checks, and guarded critical-path dependencies.
  • Quality posture: frontend Vitest coverage, backend OpenAPI contract assertions, and split-deployment smoke release checks.
  • Observability posture: correlation IDs in headers/envelopes, structured request logs, and threshold-driven runtime metrics alerts.
  • Incident response posture: class-based runbooks for auth, migration, restore, CORS, and SSE failures are documented.
  • Trip lifecycle, file handling, and data flow are documented as core business rules.
  • Production deployment is split between frontend-next hosting and backend API hosting.
