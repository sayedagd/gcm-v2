Architecture

Split system architecture

GCM v2 runs as two independently deployed applications: a Next.js 16 frontend and an API-only backend. They operate as one product through a versioned HTTP contract and explicit environment-based integration.

Top Level

Runtime boundaries

  • Frontend project: frontend-next/ on Vercel, Next.js 16 App Router.
  • Backend project: backend/ on Vercel, Express API routed through api/[...path].js.
  • Legacy frontend: frontend/ is decommissioned and blocked by default scripts.
  • App documentation site: app-documentation/ for local developer docs only.
  • Database: remote PostgreSQL accessed by backend via DATABASE_URL.

Request Flow

How frontend and backend act as one app

  1. Browser loads frontend routes from the Next.js App Router deployment.
  2. Frontend data calls use NEXT_PUBLIC_API_BASE_URL (or server-side API_BASE_URL).
  3. Backend handles versioned business modules via /api/v1/* and keeps /api/* legacy aliases with deprecation headers.
  4. Backend validates JWT and role access, then performs PostgreSQL queries.
  5. Responses return to frontend for rendering and state updates.

Hardening Status

Completed runtime safeguards

  • Uploads and backups use S3-compatible object storage with signed-read URLs in production paths.
  • In-process cron is disabled by default; managed scheduler endpoints are idempotent and advisory-lock protected.
  • Heavy backend capabilities are lazy-loaded to keep API hot paths fast on serverless runtimes.
