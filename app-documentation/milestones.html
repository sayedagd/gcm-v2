# Server-Aware Data Access Strategy (P4.6)

## Objective
Define and implement explicit Next.js App Router data policies by route domain so rendering and caching behavior are predictable.

## Policy Summary
- Public marketing routes: cacheable with ISR where appropriate.
- Authenticated portals: dynamic per request with no-store fetches.
- Internal operations routes: dynamic per request with no-store fetches.
- Shared utility: server fetch wrapper enforces strategy at call site.

## Implemented Controls
- Public landing: ISR enabled with revalidate 600 seconds.
  - File: src/app/(public)/landing/page.tsx
- Client group: dynamic + no-store enforced at layout boundary.
  - File: src/app/(client)/layout.tsx
- Subcontractor group: dynamic + no-store enforced at layout boundary.
  - File: src/app/(subcontractor)/layout.tsx
- Driver group: dynamic + no-store enforced at layout boundary.
  - File: src/app/(driver)/layout.tsx
- Internal group: dynamic + no-store enforced at layout boundary.
  - File: src/app/(internal)/layout.tsx
- Shared server fetch helper for strategy-specific API calls.
  - File: src/lib/serverFetch.ts

## Route Matrix
| Route Domain | Example Paths | Rendering Mode | Fetch Policy |
|---|---|---|---|
| Public | /landing, /home, /login | Static/ISR | force-cache or revalidate |
| Client Portal | /client/* | Dynamic | no-store |
| Subcontractor Portal | /subcontractor/* | Dynamic | no-store |
| Driver Portal | /driver/* | Dynamic | no-store |
| Internal Ops/Admin | /db, /t, /rd, /sys, etc. | Dynamic | no-store |

## Usage Guideline
When adding data access to migrated pages, use src/lib/serverFetch.ts and set strategy based on route domain:
- Public informational data: revalidate/static
- User/session scoped or frequently mutating data: dynamic

# Incident Class Runbooks (P9.3)

## Scope
Operational response playbooks for critical incident classes in split frontend/backend deployment.

## 1) Auth Outage
### Symptoms
- Repeated 401 responses on protected routes.
- Surge in `AUTH_TOKEN_EXPIRED` or `AUTH_TOKEN_FAILED`.
- Login endpoint failures.

### Immediate Actions
1. Verify `JWT_SECRET`, `JWT_EXPIRES_IN`, and `AUTH_COOKIE_*` values are set correctly in backend runtime.
2. Check `/api/v1/system/metrics` for auth failure spike.
3. Confirm frontend/backend clock drift is not causing premature expiry.
4. Validate `/api/v1/auth/login` and `/api/v1/auth/sse-token` manually.

### Stabilization
- Roll back recent auth config changes.
- Temporarily lower login throttling only if needed and monitored.

## 2) DB Migration Failure
### Symptoms
- Startup migration logs report schema drift or SQL errors.
- CRUD endpoints fail with relation/column errors.

### Immediate Actions
1. Run schema health endpoint (`/api/v1/system/schema-health`) as admin.
2. Compare missing/extra columns against canonical schema and migration scripts.
3. Stop new destructive operations until schema parity is restored.

### Stabilization
- Apply verified migration SQL in staging first, then production.
- Re-run health checks and OpenAPI contract tests.

## 3) Backup Restore Failure
### Symptoms
- `/api/v1/system/backup/restore` returns 500.
- Restore job terminates before DB import completion.

### Immediate Actions
1. Verify uploaded file type and integrity.
2. Inspect backend logs + metrics (`backupFailures24h`).
3. Confirm object storage accessibility and DB write permissions.

### Stabilization
- Retry with known-good artifact from previous successful backup.
- If repeated failure: pause restore attempts and escalate with artifact hash + error trace IDs.

## 4) CORS Failure
### Symptoms
- Browser rejects API calls with CORS errors.
- Preflight failures after environment/domain changes.

### Immediate Actions
1. Verify `CORS_ORIGIN` and `CORS_ALLOWED_ORIGINS` values.
2. Confirm request origin exactly matches allowlisted domain/protocol.
3. Validate credentials mode and cookie flags (`sameSite`, `secure`).

### Stabilization
- Apply explicit allowlist fix, avoid wildcard credentials policies.
- Re-test from production and preview origins.

## 5) SSE Failure
### Symptoms
- Event stream not connecting or frequent disconnects.
- Missing real-time updates in client dashboards.

### Immediate Actions
1. Verify `/api/v1/events` auth flow and SSE token issuance endpoint.
2. Check `sseDisconnects15m` metric and recent alert entries.
3. Inspect reverse proxy buffering/timeouts and heartbeat behavior.

### Stabilization
- Ensure `X-Accel-Buffering: no` behavior is preserved in deployed proxy.
- Validate client reconnect/backoff and token refresh behavior.

## Cross-Incident Checklist
1. Capture `traceId` from error envelope or `X-Correlation-Id` header.
2. Record incident timeline and impacted route groups.
3. Verify smoke suite (`scripts/smoke/split-deployment-smoke.mjs`) before declaring recovery.
4. Add postmortem action items and threshold tuning updates.

# Route Render and Cache Matrix (P7.1)

## Purpose
Define route-level rendering and cache behavior for predictable performance on web and mobile.

## Matrix

| Route family | Rendering mode | Cache policy | Invalidation trigger |
| --- | --- | --- | --- |
| `/landing` | ISR | `revalidate = 600` | Automatic 10-minute refresh; manual deploy refresh |
| `/` | Static redirect | `force-static` | New deploy only |
| `/home` | Static redirect | `force-static` | New deploy only |
| `/login` | Dynamic | `force-no-store` | Always fresh (query/session-sensitive) |
| `/logout` | Dynamic | `force-no-store` | Always fresh (session mutation) |
| `/unauthorized` | Static | `force-static` | New deploy only |
| `/(client)/*` | Dynamic | `force-no-store` via group layout | Always fresh (auth + operational data) |
| `/(subcontractor)/*` | Dynamic | `force-no-store` via group layout | Always fresh (auth + operational data) |
| `/(driver)/*` | Dynamic | `force-no-store` via group layout | Always fresh (auth + operational data) |
| `/(internal)/*` | Dynamic | `force-no-store` via group layout | Always fresh (auth + operational data) |

## Implementation Anchors
- Public ISR route: `frontend-next/src/app/(public)/landing/page.tsx`
- Public auth routes: `frontend-next/src/app/(public)/login/page.tsx`, `frontend-next/src/app/(public)/logout/page.tsx`
- Static public pages: `frontend-next/src/app/page.tsx`, `frontend-next/src/app/(public)/home/page.tsx`, `frontend-next/src/app/(public)/unauthorized/page.tsx`
- Dynamic portal layouts:
  - `frontend-next/src/app/(client)/layout.tsx`
  - `frontend-next/src/app/(subcontractor)/layout.tsx`
  - `frontend-next/src/app/(driver)/layout.tsx`
  - `frontend-next/src/app/(internal)/layout.tsx`

## Notes
- This matrix intentionally keeps authenticated areas uncached at the Next.js fetch layer to avoid stale operational state.
- Future optimization in P7.2/P7.3 can selectively introduce tagged revalidation for high-read/low-write slices.

# Split Deployment Smoke Suite (P8.3)

## Script
- `scripts/smoke/split-deployment-smoke.mjs`

## Required Environment
- `SMOKE_FRONTEND_BASE_URL` (for example `https://frontend.example.com`)
- `SMOKE_BACKEND_BASE_URL` (for example `https://api.example.com`)
- Optional: `SMOKE_TIMEOUT_MS` (default `10000`)
- Optional: `SMOKE_DRY_RUN=true` to validate configured checks without network calls.

## Checks
1. Frontend landing route is reachable.
2. Backend ping route is reachable.
3. Frontend dashboard route is reachable or redirects (auth boundary check).
4. Invalid login returns `401`.
5. Key CRUD write surface (`POST /api/v1/companies`) rejects anonymous access (`401`).
6. Admin backup status endpoint rejects anonymous access with canonical `AUTH_NO_TOKEN` envelope.
7. SSE endpoint health (`/api/v1/events`) rejects anonymous access (`401`).

## Usage
```bash
SMOKE_FRONTEND_BASE_URL=https://frontend.example.com \
SMOKE_BACKEND_BASE_URL=https://api.example.com \
```
