Operational playbook

Use these steps for backup, restore, release, incident response, and rollback. They are written for a separate frontend and backend deployment model.

Home Overview Backend Schema Environment Print export
  • Backup: use backend backup service; production backups are persisted as object-storage artifacts and served via signed URL downloads.
  • Restore: use the restore endpoint with a known-good backup artifact and confirm schema health afterward.
  • Release: deploy backend API separately from frontend-next (Next.js) project.
  • Frontend performance gate: run npm run ci:perf in frontend-next/ before release approvals.
  • Split-deployment smoke gate: run node scripts/smoke/split-deployment-smoke.mjs with SMOKE_FRONTEND_BASE_URL and SMOKE_BACKEND_BASE_URL before release sign-off.
  • Cutover checklist gate: run node scripts/release/cutover-checklist.mjs with CUTOVER_FRONTEND_BASE_URL, CUTOVER_BACKEND_BASE_URL, and CUTOVER_FALLBACK_FRONTEND_URL.
  • Cutover stability watch: run node scripts/release/cutover-watch.mjs during the release window to sample error-rate and latency.
  • Incident response: check schema health, event stream, logs, auth expiry codes (AUTH_TOKEN_EXPIRED), and 429 spikes by endpoint tier.
  • Rollback: redeploy the previous frontend-next build and roll back backend only when migration compatibility allows it.

Scheduler operations

  • In-process cron is disabled by default (ENABLE_IN_PROCESS_CRON=false).
  • Run scheduled backups from managed scheduler jobs that call authenticated system endpoints.
  • Use idempotency keys for scheduler-triggered backup routes to avoid duplicate runs.
  • Monitor backup_job_runs and backup_artifacts tables for execution/audit status.

Backend maintenance operations

Run maintenance commands only from backend/, against the intended environment, after taking a fresh backup.

  • npm run maintenance:create-test-accounts: use to create or refresh standard role-based test users in non-production environments.
  • npm run maintenance:inject-services: use to sync baseline service and facility definitions after seed/schema-aligned updates.
  • npm run maintenance:reset-admin: use only for account recovery when the primary admin cannot access the system.
  • npm run maintenance:reset-testing: destructive test reset; wipes transactional data while preserving service tree and one admin.
  • npm run maintenance:wipe-data: high-impact wipe of operational data while preserving users/services/config.
Safety checks: confirm target database, verify DATABASE_URL, run in a maintenance window, and announce completion in ops notes. Do not run destructive commands on production without explicit approval and a restore plan.

Auth and rate-limit operations

  • Token TTL and cookie behavior are controlled by JWT_EXPIRES_IN and AUTH_COOKIE_* variables.
  • If users are unexpectedly redirected to login, inspect backend 401 envelopes for AUTH_TOKEN_EXPIRED.
  • 429 incidents should be triaged by route tier: auth, public-write, SSE token issuance, AI, or admin-sensitive operations.
  • Adjust only the relevant RL_* variable for the affected tier instead of increasing global limits.

Archived script decision matrix

  • Keep archived: script is idempotent, uses current DB bootstrap patterns, and may be needed for legacy environment recovery.
  • Archive-delete candidate: script uses hardcoded local credentials, Docker-local assumptions, or schema changes that no longer match canonical SQL.
  • Delete (only after hold period): no npm alias, no runtime/docs references, and columns/tables are verified as already covered or obsolete.

Current recommendation: keep fix-trips-columns.js archived for conservative recovery. manual-migration.js was removed after classification as an archive-delete candidate due to hardcoded localhost credentials and legacy one-off SQL.

Incident Class Runbooks

Detailed runbook source is maintained in the frozen milestones reference page (milestones.html). Quick-response procedures are summarized here for on-call usage.

  • Auth outage
  • DB migration failure
  • Backup restore failure
  • CORS failure
  • SSE failure

Auth outage

  • Validate JWT_SECRET, JWT_EXPIRES_IN, and AUTH_COOKIE_* runtime values.
  • Inspect /api/v1/system/metrics for auth failure spikes.
  • Verify /api/v1/auth/login and /api/v1/auth/sse-token responses.

DB migration failure

  • Run /api/v1/system/schema-health and identify drift.
  • Pause destructive writes until schema parity is restored.
  • Apply verified migration SQL in staging before production.

Backup restore failure

  • Validate uploaded artifact integrity and content type.
  • Inspect logs and backupFailures24h metric.
  • Retry with last known-good backup artifact only.

CORS failure

  • Verify CORS_ORIGIN and CORS_ALLOWED_ORIGINS are exact and protocol-correct.
  • Confirm credential mode and cookie flag compatibility.
  • Re-test from production and preview origins after updates.

SSE failure

  • Verify stream auth and SSE token issuance flow.
  • Inspect sseDisconnects15m metric and active alerts.
  • Check proxy buffering/timeouts and heartbeat continuity.
