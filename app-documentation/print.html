Overview

GCM v2 is an environmental operations platform for companies that collect projects, recyclables, and waste. The core system manages companies, projects, services, trips, drivers, vehicles, suppliers, facilities, inventory, reports, AI sessions, and public landing/store flows.

Frontend

The active frontend is frontend-next/ (Next.js 16 App Router + React 19). It uses route groups, middleware role guards, server-aware data strategy, React action patterns, and route-level loading boundaries. Feature modules now follow a strict ui/hooks/api/model/tests contract enforced by lint-time checks. Legacy frontend/ (Vite) is decommissioned. Performance governance now includes an explicit route render/cache matrix, bundle budget scripts, critical dependency guardrails, and Next image optimization with lazy media delivery.

Backend

The backend is an API-only Express service. It owns auth, route modules, migrations, backups, event stream handling, WhatsApp, OCR, and AI services. Public API contract is versioned under /api/v1 with OpenAPI source in backend/openapi/openapi.v1.json. Security hardening includes global write-request sanitization/body-shape checks, token-expiry re-auth contract, HttpOnly cookie auth lifecycle controls, and endpoint-tier rate limiting. Observability includes request correlation IDs, structured request logs, and threshold-based metrics exposed via admin system endpoints.

Schema

The database contract centers on users, companies, projects, services, trips, logistics tables, system config, and reporting/AI history. Schema alignment was normalized by removing duplicate DDL and aligning runtime contract names.

API

Core route groups include auth, trips, projects, companies, services, vehicles, drivers, facilities, suppliers, and system backup/config endpoints under /api/v1. Frontend endpoint constants and typed operation wrappers are centralized in core feature modules to reduce contract drift risk. Legacy /api/* aliases remain temporarily with deprecation headers.

User flows

Admin, operations staff, client, subcontractor, and driver journeys are separated by role so each persona sees an appropriate slice of the platform.

Runbooks

Backup, restore, release, incident response, and rollback steps are documented for split deployment. Backup artifacts are object-storage-backed; in-process cron is disabled by default and scheduler calls are idempotent/lock-protected. Release quality gates include frontend unit tests, backend OpenAPI contract checks, and split-deployment smoke validation.
