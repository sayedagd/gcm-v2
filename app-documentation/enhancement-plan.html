Force Required Procedural

Execution rules used during implementation

  1. Follow plan order strictly; do not defer or skip tasks.
  2. Execute one task at a time and mark task status as in progress before coding.
  3. After each task completion, run validation: typecheck, lint, touched-area tests, and build-check where needed.
  4. If validation fails, fix immediately before moving to next task.
  5. Only after clean validation, update task status to completed.
  6. Start next task automatically without asking user.
  7. Keep user replies minimal during execution.
  8. Never use static or hardcoded app data for business entities; all app data must be DB-driven.
  9. Keep migrations, schema, and API contracts aligned at every task.
  10. Do not close a phase until all success criteria pass.

Goal

Program objectives

  • Fix all security, contract, and runtime gaps from architecture review.
  • Enforce strict frontend/backend decoupling.
  • Migrate frontend from Vite SPA to Next.js 16 App Router under src/app.
  • Keep split deployment model (frontend project and backend project) on Vercel.

Phase 0

Program setup and control

  • P0.1 Establish migration branch strategy [not started]: create long-lived migration branch, define feature branch naming, and set merge/rollback checklist.
  • P0.2 Define architecture decision records [not started]: ADRs for split deploy model, API-only backend, Next.js migration, storage model, and auth model.
  • P0.3 Baseline architecture audit [completed]: repository-wide review completed and findings mapped into this plan.

Phase 1

Critical security and contract fixes

  • P1.1 Remove frontend secret exposure [completed]: removed injected API key definitions from frontend build config and verified no secret leakage in assets.
  • P1.2 Eliminate client-controlled validation bypass [completed]: removed x-skip-validation behavior from public API paths.
  • P1.3 Fix backup endpoint contract drift [completed]: aligned frontend backup endpoints with backend route mounts and added contract checks.
  • P1.4 Harden CORS preview policy [completed]: replaced wildcard preview suffix logic with explicit allowlist.
  • P1.5 Improve SSE auth transport [completed]: removed long-lived query-token model and applied safer auth transport.

Phase 2

Strict decoupling and API governance

  • P2.1 Introduce versioned API namespace [completed]: public contracts standardized under versioned strategy.
  • P2.2 Publish OpenAPI spec [completed]: core modules documented with request/response and error schemas.
  • P2.3 Generate typed API client [completed]: code generation workflow added and manual endpoint drift reduced.
  • P2.4 Unify validation source of truth [completed]: backend retained as authoritative validation layer.
  • P2.5 Standardize API error contract [completed]: canonical envelope enforced across controllers.

Phase 3

Backend runtime refactor for Vercel reliability

  • P3.1 Separate startup side effects [completed]: heavy startup work moved out of hot request bootstrap paths.
  • P3.2 Decouple heavy capabilities [completed]: Puppeteer/WhatsApp-heavy paths isolated from core CRUD/auth runtime.
  • P3.3 Replace local backup storage [completed]: backups moved to durable object storage with DB metadata and signed access.
  • P3.4 Replace local uploads model [completed]: upload flows moved to object storage for serverless-safe persistence.
  • P3.5 Replace in-process cron [completed]: managed scheduler model with idempotency and lock protection implemented.

  • P4.4 Migrate auth/session boundary [completed].
  • P4.5 Migrate pages domain-by-domain [completed].

  • P4.7 Introduce React 19 action patterns [completed].

Definition Of Done

Program-level completion criteria

  • Strict decoupling enforced: frontend can be replaced without backend contract changes.
  • Backend API contract versioned, documented, and test-enforced.
  • Frontend runs on Next.js 16 App Router under src/app with feature parity.
  • Serverless-hostile local persistence and in-process scheduling removed from production path.
  • Security-critical gaps closed: secrets, auth transport, validation bypass, and CORS hardening.
  • Observability, test gates, and runbooks in place for reliable operations.
