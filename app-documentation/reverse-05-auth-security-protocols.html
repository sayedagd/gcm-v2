# 05. Auth & Security Protocols

## 1. Authentication Mechanisms

## 1.1 Backend API Authentication

- JWT signing algorithm: `HS256`
- Token claims: `id`, `role`, `company_id`, `project_id`, `supplier_id`
- Token lifetime: `JWT_EXPIRES_IN` (default controller behavior targets 12h)
- Token transport:
  - Bearer token in `Authorization` header
  - HttpOnly cookie (default cookie name `gcm_jwt`)
  - Query token only for SSE endpoint (`/api/events`) and only with `purpose=sse`

## 1.2 Frontend Session Layer (Current Migration Mode)

- Frontend uses server actions to set cookies:
  - `gcm_auth_session`
  - `gcm_current_role`
  - `gcm_auth_exp`
- This frontend mode currently validates role selection locally and does not call backend credential auth.

## 2. Authorization Model

## 2.1 API Authorization

- `protect` middleware required on private route groups.
- `authorizeRoles('ADMIN')` applied to high-risk system endpoints:
  - backup
  - health
  - metrics
  - schema operations
  - whatsapp QR

## 2.2 Frontend Route Authorization

Role-path policy examples:

- `/client/*` -> `CLIENT`, `COMPANY_USER`, `PROJECT_USER`
- `/subcontractor/*` -> `SUBCONTRACTOR`
- `/driver/*` -> `DRIVER`
- `/rd/*` -> `ADMIN`, `REPORTS_MANAGER`
- `/acc/*`, `/st/*`, `/sys/*` etc. -> role-specific boundaries

## 2.3 SSE Visibility Authorization

SSE event visibility is filtered per role and ownership context:

- Driver: only own driver events
- Subcontractor: supplier-scoped events
- Client/company/project users: scoped company/project visibility
- Internal staff roles: broad visibility

## 3. Security Middleware Stack

- `helmet` with tuned CSP directives
- `cors` allowlist gate with explicit origin checks
- Request context/correlation id middleware
- Structured rate limiting profiles
- Write payload shape validation and input sanitization
- Centralized error handler with canonical envelope

## 4. Rate Limit Profiles

Configurable by env vars:

- Global limiter
- Auth limiter (login brute-force protection)
- Public write limiter (contact/store submissions)
- SSE token limiter
- AI request limiter
- Admin operation limiter

## 5. Error Envelope Standard

Canonical payload includes:

- `status`
- `code`
- `error`
- `errorEn`
- `errorAr`
- `traceId`
- Optional `field` and `details`

This supports both legacy frontend compatibility and modern tracing.

## 6. Operational Security Controls

- Correlation headers propagated (`X-Correlation-Id`)
- Request metrics and threshold-based alerts
- Auth failure counters
- Backup failure counters
- SSE disconnect tracking

## 7. Security Risks Observed (Current State)

- Backend login currently compares plain-text password values in controller.
- Frontend migration login bypasses backend credential verification.
- Some config routes are currently public and should be reviewed for intended exposure.
- `getPing` diagnostic includes sensitive debug account checks and should be constrained in production.

## 8. Hardening Recommendations

1. Move all login validation to bcrypt-based backend verification.
2. Retire migration role-only frontend session flow.
3. Restrict diagnostics/config endpoints by role or signed internal access.
4. Add explicit token revocation/session invalidation strategy.
5. Add automated security tests for auth/cors/rate-limit bypass attempts.
