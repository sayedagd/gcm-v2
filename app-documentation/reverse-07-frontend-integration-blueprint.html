# 07. Frontend Integration Blueprint (API Consumption)

## 1. Goal

Enable a new frontend team to connect to the existing backend API quickly and safely while preserving auth, tracing, and versioning conventions.

## 2. Base URL and Versioning

- Use API base URL from environment:
  - `API_BASE_URL` (server)
  - `NEXT_PUBLIC_API_BASE_URL` (client when needed)
- Use versioned prefix for all new integration: `/api/v1`

Example:

- Base URL: `https://backend.example.com`
- Login endpoint: `https://backend.example.com/api/v1/auth/login`

## 3. Required Headers

For standard JSON calls:

- `Accept: application/json`
- `Content-Type: application/json` (for writes)
- `Authorization: Bearer <token>` (for protected endpoints)
- `X-Correlation-Id: <uuid>`
- `X-Request-Id: <uuid>`

For SSE stream:

- Fetch `GET /api/v1/auth/sse-token` first using normal auth.
- Connect with EventSource to `/api/v1/events?token=<short-lived-sse-token>`.

## 4. Authentication Integration Options

## Option A (Recommended)

- Perform backend login with credentials.
- Receive JWT and backend auth cookie.
- Store token only in memory when possible.
- Send bearer token for API calls.

## Option B (Cookie-Centric)

- Rely primarily on HttpOnly backend auth cookie.
- Ensure CORS and cookie flags align across domains.

## 5. Error Handling Contract

All frontend API layers should parse canonical envelope fields:

- `code`
- `error`
- `errorEn`
- `errorAr`
- `traceId`
- `field` (optional)
- `details` (optional)

UI should display friendly localized message and include `traceId` in support workflows.

## 6. Core Data Structures (Observed)

## 6.1 User

- `id`, `name`, `email`, `role`, optional tenant-scope ids (`company_id`, `project_id`, `supplier_id`)

## 6.2 Company

- `company_id`, `company_name`, compliance and contact metadata

## 6.3 Project

- `project_id`, `project_name`, `company_id`, schedule, budget, asset counts

## 6.4 Trip

- `trip_id`, `project_id`, `service_id`, `vehicle_id`, `driver_id`, `facility_id`
- status/timestamps
- compliance identifiers and files
- approval/signature fields

## 6.5 Notifications

- `id`, `title`, `message`, `type`, `read`, `timestamp`, `user_id`

## 7. Suggested Frontend Service Layer

1. Create one API client module that injects auth + correlation headers.
2. Create domain service modules by backend bounded context:
   - auth
   - core
   - operations
   - logistics
   - reporting
   - infra/admin
3. Keep endpoint constants in a central contract file.
4. Normalize errors into one UI-consumable shape.

## 8. Migration Plan for New Frontend Team

1. Import endpoint constants and define typed request/response interfaces.
2. Integrate real backend login/logout first.
3. Replace temporary role-only session cookies with backend auth flow.
4. Implement protected-route guard from decoded role or backend profile endpoint.
5. Add SSE subscription service with reconnection and token refresh.
6. Add smoke tests for critical journeys:
   - login/logout
   - list/update core entities
   - trip create/update with validation handling
   - backup status authorization checks

## 9. Compatibility Notes

- Legacy routes still exist but are sunset-marked; do not integrate new screens against `/api/*`.
- Prefer `/api/v1/*` exclusively for new frontend architecture.
